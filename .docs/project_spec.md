## 1. Project Overview

**Emitkit** is an open-source SDK generator platform. Admins connect a GitHub repository (existing or newly created), point to an OpenAPI spec, select desired outputs, and Emitkit generates and commits the outputs as a pull request.

### Outputs Supported

| Output     | Languages          | Description                                                            |
| ---------- | ------------------ | ---------------------------------------------------------------------- |
| SDK        | TypeScript, Python | Typed fetch client (TS) / httpx client (Python), one file per resource |
| CLI        | TypeScript         | commander.js CLI, one command per operation                            |
| MCP Server | TypeScript         | `@modelcontextprotocol/sdk` server, one tool per operation             |
| Docs       | Markdown           | API reference, optionally enriched by Gemini                           |

### Core User Flow

```
GitHub OAuth
  ŌåÆ Create Project
      ŌåÆ Option A: connect existing repo + specify spec path
      ŌåÆ Option B: create new GitHub repo (Emitkit creates it) + upload/specify spec
  ŌåÆ Configure outputs + languages + optional Gemini key
  ŌåÆ Trigger generation (manual or auto on spec file change via webhook)
  ŌåÆ Emitkit runs syntax checks + formatting on generated files
  ŌåÆ Emitkit commits output files to a new branch + opens PR
  ŌåÆ Admin reviews and merges
  ŌåÆ On merge: GitHub Actions (generated by Emitkit) publish SDK to npm / PyPI
```

## 3. Features & Requirements

### 3.1 Authentication

- GitHub OAuth via Better Auth
- Users belong to one or more organizations
- Organization-level access: any org member can access all org projects
- Session stored in SQLite via Better Auth adapter
- All oRPC procedures require authenticated session; unauthenticated requests return `401`

### 3.2 Project Management

**Connecting an existing repo:**

- Admin selects a GitHub repo they have write access to
- Admin specifies the path to the OpenAPI spec file within the repo (e.g. `openapi.yaml`, `docs/api.json`)
- Admin selects the default branch

**Creating a new repo:**

- Admin provides a repo name + visibility (public/private)
- Emitkit creates the repo via GitHub API under the authed user or org
- Emitkit creates an initial commit with a starter `openapi.yaml` stub
- Project is then configured normally from there

**Common rules:**

- One repo = one project (enforced at DB level with `unique(orgId, repoFullName)`)
- Project can be deleted; deletion cascades to configs and runs
- Output mode: `append` (default) ŌĆö outputs committed into the same repo; or `separate` ŌĆö outputs committed to a dedicated output repo (configured per project)

### 3.3 Project Configuration

- Select one or more outputs: `SDK`, `CLI`, `MCP`, `DOCS`
- Select SDK languages: `typescript`, `python` (multi-select; at least one required if SDK selected)
- Output directory within repo (default: `.emitkit/`)
- Output mode: `append` (same repo) | `separate` (new dedicated repo, auto-created on first run)
- SDK: optional npm scope (e.g. `@acme`) for TypeScript; optional PyPI package name for Python
- SDK version strategy: `emitkit-managed` (default) | `spec-version` (read from `info.version`)
- Docs: optional Gemini API key (stored encrypted at rest)
- Config is versioned ŌĆö each save creates a new config snapshot (simple append, query latest by `createdAt DESC`)

### 3.4 Generation Runs

- Manual trigger via dashboard
- Automatic trigger via GitHub webhook on `push` event if the spec file path is in the changed files
- Each run records: status (`queued` | `running` | `success` | `failed`), logs, commit SHA, PR URL, SDK version bumped to
- Logs are streamed in real time via SSE on the run detail page
- Runs are immutable once finished

### 3.5 SDK Publishing

- Emitkit generates GitHub Actions workflow files as part of each run output
- `publish-npm.yml` ŌĆö triggers on PR merge to default branch, publishes TypeScript SDK to npm
- `publish-pypi.yml` ŌĆö triggers on PR merge, publishes Python SDK to PyPI via OIDC trusted publishing
- Emitkit does **not** hold npm tokens or PyPI credentials; publishing is delegated to the repo's CI
- Admin configures npm/PyPI trusted publishing in their repo settings once; Emitkit generates the workflow

### 3.6 Webhook

- `POST /webhooks/github` ŌĆö plain Hono route (not oRPC)
- Verifies GitHub HMAC signature (`X-Hub-Signature-256`)
- Filters `push` events where changed files include the configured spec path
- Also listens for `pull_request` closed + merged events to trigger post-merge tagging
- Enqueues a BullMQ job; responds `200 OK` immediately

## 4. Architecture

### Request Flow (oRPC)

```
Browser ŌåÆ React (oRPC client)
       ŌåÆ Hono server ŌåÆ oRPC handler
       ŌåÆ procedure ŌåÆ service layer
       ŌåÆ DB (Drizzle/libSQL)
       ŌåÆ BullMQ (enqueue jobs)
       ŌåÆ GitHub API (Octokit)
```

### Generation Run Flow

```
trigger (manual or webhook push)
  ŌåÆ insert GenerationRun (status: queued)
  ŌåÆ enqueue BullMQ job: { runId }
  ŌåÆ worker picks up job
      ŌåÆ fetch spec from GitHub at commit SHA
      ŌåÆ parse spec ŌåÆ IR (ParsedSpec)
      ŌåÆ diff against last successful run's spec ŌåÆ detect breaking changes
      ŌåÆ calculate new SDK version
      ŌåÆ run selected generators in parallel
          ŌåÆ SDK generator (TS)   ŌåÆ files[] + syntax check (tsc) + format (prettier)
          ŌåÆ SDK generator (Py)   ŌåÆ files[] + lint (ruff) + format (ruff format)
          ŌåÆ CLI generator        ŌåÆ files[] + syntax check + format
          ŌåÆ MCP generator        ŌåÆ files[] + syntax check + format
          ŌåÆ Docs generator       ŌåÆ files[] (+ optional Gemini calls)
      ŌåÆ generate CI workflow files (publish-npm.yml, publish-pypi.yml)
      ŌåÆ collect all output files (skip errored generators, log warnings)
      ŌåÆ resolve output repo (same repo or separate repo)
      ŌåÆ create branch `emitkit/run-{runId}` on output repo
      ŌåÆ commit files via GitHub Tree API
      ŌåÆ open Pull Request
      ŌåÆ update GenerationRun (status: success, pr_url, sdkVersion)
  ŌåÆ SSE stream closes
```

### BullMQ Architecture

```
apps/server   ŌåÆ enqueues jobs to Redis via BullMQ
apps/worker   ŌåÆ separate Bun process, consumes jobs from Redis
              ŌåÆ one Worker for queue: 'generation'
              ŌåÆ concurrency: 3 (configurable via env)
              ŌåÆ job timeout: 5 minutes
              ŌåÆ retries: 2 (on unexpected crash only, not on business logic failure)
```

- Server and worker share `packages/queue` for queue name constants and job type definitions
- Worker writes logs to DB and publishes SSE events via Redis pub/sub ŌåÆ server reads and streams to browser
- On worker startup: query all `running` runs and mark them `failed` with log: `"Worker restarted; run aborted"`

### SSE Log Streaming via Redis Pub/Sub

```
worker ŌåÆ publishes log lines to Redis channel: `run-logs:{runId}`
server SSE route ŌåÆ subscribes to Redis channel ŌåÆ streams to browser
```

This decouples the worker process from the HTTP server cleanly.

## 5. Data Model

All tables use Drizzle ORM with libSQL. Primary keys are `cuid2` strings.

### users

```ts
{
  id: text primaryKey,
  githubId: text unique notNull,
  email: text unique notNull,
  name: text notNull,
  avatarUrl: text,
  accessToken: text notNull,   // GitHub OAuth token, stored encrypted
  createdAt: integer notNull,  // Unix timestamp
}
```

### organizations

```ts
{
  id: text primaryKey,
  githubOrgId: text unique notNull,
  name: text notNull,
  slug: text unique notNull,
  createdAt: integer notNull,
}
```

### organization_members

```ts
{
  orgId: text notNull references organizations.id,
  userId: text notNull references users.id,
  role: text notNull default 'member',   // 'owner' | 'member'
  primary key (orgId, userId),
}
```

### projects

```ts
{
  id: text primaryKey,
  orgId: text notNull references organizations.id,
  repoFullName: text notNull,        // e.g. 'acme/api'
  specPath: text notNull,            // e.g. 'openapi.yaml'
  defaultBranch: text notNull default 'main',
  outputMode: text notNull default 'append',   // 'append' | 'separate'
  outputRepoFullName: text,          // set if outputMode = 'separate'
  webhookId: integer,
  webhookSecret: text,               // HMAC secret, stored encrypted
  createdAt: integer notNull,
  unique (orgId, repoFullName),
}
```

### project_configs

```ts
{
  id: text primaryKey,
  projectId: text notNull references projects.id,
  outputs: text notNull,             // JSON array: ['SDK','CLI','MCP','DOCS']
  sdkLanguages: text notNull,        // JSON array: ['typescript','python']
  outputDir: text notNull default '.emitkit/',
  sdkNpmScope: text,                 // e.g. '@acme'
  sdkPypiName: text,                 // e.g. 'acme-sdk'
  sdkVersionStrategy: text notNull default 'emitkit-managed',
  geminiApiKey: text,                // encrypted, nullable
  createdAt: integer notNull,
}
```

> Only the latest config per project is "active". Query with `ORDER BY createdAt DESC LIMIT 1`.

### generation_runs

```ts
{
  id: text primaryKey,
  projectId: text notNull references projects.id,
  configId: text notNull references project_configs.id,
  triggeredBy: text notNull,         // 'manual' | 'webhook'
  status: text notNull default 'queued',  // 'queued'|'running'|'success'|'failed'
  commitSha: text,                   // spec commit SHA that triggered this run
  specSnapshot: text,                // JSON snapshot of ParsedSpec (for diffing next run)
  sdkVersion: text,                  // e.g. '1.2.0' ŌĆö version bumped in this run
  branchName: text,
  prUrl: text,
  logs: text,                        // append-only newline-delimited log string
  createdAt: integer notNull,
  finishedAt: integer,
}
```

### sdk_versions

```ts
{
  id: text primaryKey,
  projectId: text notNull references projects.id,
  version: text notNull,             // semver e.g. '1.2.0'
  runId: text notNull references generation_runs.id,
  changeType: text notNull,          // 'major' | 'minor' | 'patch'
  changelog: text,                   // auto-generated markdown changelog
  createdAt: integer notNull,
}
```

```
  Ōö£ŌöĆŌöĆ projects
  Ōöé   Ōö£ŌöĆŌöĆ list(orgId)                   ŌåÆ Project[]
  Ōöé   Ōö£ŌöĆŌöĆ get(projectId)                ŌåÆ Project
  Ōöé   Ōö£ŌöĆŌöĆ createFromExistingRepo(input) ŌåÆ Project
  Ōöé   Ōö£ŌöĆŌöĆ createNewRepo(input)          ŌåÆ Project

  Ōöé   ŌööŌöĆŌöĆ config

  Ōöé       ŌööŌöĆŌöĆ save(input)              ŌåÆ ProjectConfig
  Ōö£ŌöĆŌöĆ runs
  Ōöé   Ōö£ŌöĆŌöĆ list(projectId)               ŌåÆ Run[]
  Ōöé   Ōö£ŌöĆŌöĆ get(runId)                    ŌåÆ Run
  Ōöé   ŌööŌöĆŌöĆ trigger(projectId)            ŌåÆ Run
  ŌööŌöĆŌöĆ versions
      ŌööŌöĆŌöĆ list(projectId)               ŌåÆ SdkVersion[]
```

### Plain Hono Routes (not oRPC)

```
POST /webhooks/github              ŌåÆ GitHub push + PR merged event handler
GET  /runs/:runId/logs/stream      ŌåÆ SSE log stream (reads from Redis pub/sub)
```

### SSE Log Stream

- Client subscribes to `/runs/:runId/logs/stream`
- Server subscribes to Redis channel `run-logs:{runId}` and pipes messages as SSE
- Format: `data: <log line>\n\n`
- On run completion, worker publishes `[DONE]` to channel; server forwards and closes SSE
- Client reconnects automatically on disconnect (EventSource default)

## 7. Generator Modules

All generators share this contract:

```ts
interface Generator {
  generate(spec: ParsedSpec, config: GeneratorConfig): Promise<GeneratorResult>;
}

interface GeneratorResult {
  files: OutputFile[];
  error?: string; // set if generator failed; files may be partial
}

interface OutputFile {
  path: string; // relative to outputDir, e.g. 'sdk/ts/index.ts'
  content: string;
  skipIfExists?: boolean; // true for custom/ stub files ŌĆö never overwrite
}

interface ParsedSpec {
  info: { title: string; version: string; description?: string };
  servers: string[];
  operations: Operation[];
  schemas: Record<string, JsonSchema>;
  security: SecurityScheme[];
}

interface Operation {
  operationId: string;
  method: "GET" | "POST" | "PUT" | "PATCH" | "DELETE";
  path: string;
  summary?: string;
  description?: string;
  tags: string[];
  parameters: Parameter[];
  requestBody?: RequestBody;
  responses: Record<string, Response>;
  security?: SecurityRequirement[];
}
```

### 7.1 TypeScript SDK Generator

**Output directory:** `sdk/typescript/`

**Files emitted:**

```
sdk/typescript/
  index.ts           ŌåÆ barrel export
  types.ts           ŌåÆ all types from #/components/schemas
  errors.ts          ŌåÆ EmitError, EmitHttpError
  client.ts          ŌåÆ base client (fetch wrapper, auth, retries)
  resources/
    {tag}.ts         ŌåÆ generated resource class
  custom/
    {tag}.ts         ŌåÆ stub for hand-written extensions (skipIfExists: true)
  package.json       ŌåÆ with version from run
  tsconfig.json
  .github/
    workflows/
      publish-npm.yml
```

**Rules:**

- Group operations by first tag; untagged ŌåÆ `resources/default.ts`
- Each resource class named PascalCase after tag; methods named camelCase after `operationId`
- Auth: inject `Authorization` header from constructor `{ apiKey: string }` option
- Pagination: if 200 response has `next_cursor` or `next_page`, generate `paginate*` async generator
- Retries: exponential backoff, 3 attempts (1s / 2s / 4s), retry on 429 and 5xx
- Error classes: `EmitError` base, `EmitHttpError(status, body)`, exported from `errors.ts`
- No runtime dependencies ŌĆö pure fetch only
- Generated resource classes re-export everything from `custom/{tag}.ts` if that file exists

### 7.2 Python SDK Generator

**Output directory:** `sdk/python/`

**Files emitted:**

```
sdk/python/
  __init__.py
  client.py          ŌåÆ base httpx client, auth, retries
  types.py           ŌåÆ Pydantic v2 models from schemas
  errors.py          ŌåÆ EmitError, EmitHttpError
  resources/
    __init__.py
    {tag}.py         ŌåÆ generated resource class
  custom/
    __init__.py
    {tag}.py         ŌåÆ stub for hand-written extensions (skipIfExists: true)
  pyproject.toml     ŌåÆ with version from run
  .github/
    workflows/
      publish-pypi.yml
```

**Rules:**

- Use `httpx` (sync + async variants: `Client` and `AsyncClient`)
- Use Pydantic v2 models for all request/response types
- Method names snake_case after `operationId`
- Retries: `httpx` retry transport, same 3-attempt strategy as TS
- Auth: `api_key` constructor param ŌåÆ `Authorization: Bearer {key}` header
- Pagination: if response model has `next_cursor`, generate `paginate_{method}` generator
- Generated resource classes import and extend from `custom/{tag}.py` if it exists

### 7.3 CLI Generator (TypeScript only)

**Output directory:** `cli/`

**Rules:**

- Use `commander` package
- One sub-command per operation: `{tag} {operationId}`
- Path params ŌåÆ required positional args; query params ŌåÆ `--flag` options
- Request body ŌåÆ `--data <json>` string, parsed with `JSON.parse`
- Auth via `EMITKIT_API_KEY` env var (or `--api-key` flag override)
- Output: pretty-print JSON to stdout; errors to stderr, exit code 1

### 7.4 MCP Generator (TypeScript only)

**Output directory:** `mcp/`

**Rules:**

- Use `@modelcontextprotocol/sdk` `Server` class
- One MCP tool per operation
- Tool name: `operationId` (converted to snake_case)
- Tool description: `summary` or `description` from spec
- Tool input schema: JSON Schema from parameters + requestBody merged
- Tool handler: calls the generated TypeScript SDK (`../sdk/typescript`)
- Transport: `StdioServerTransport`
- Export `startServer()` as default export
- MCP server version mirrors SDK version

### 7.5 Docs Generator

**Output directory:** `docs/v{version}/`

**Structure:**

```
docs/
  v{version}/
    README.md        ŌåÆ index with links to all resources
    {tag}.md         ŌåÆ one file per tag/resource
  latest/            ŌåÆ copy of most recent version (overwritten each run)
```

**Without Gemini:** extract `summary`, `description`, parameters, request/response schemas ŌåÆ render as structured markdown with param tables and schema code blocks.

**With Gemini:** call Gemini per operation (see Section 12) to produce enriched description, examples, and edge cases. All structure (headers, param tables) remains deterministic; only prose description is LLM-generated.

## 8. Syntax Checking & Formatting

Runs in a temp directory **before** any GitHub commit. Think of it as a validate ŌåÆ format ŌåÆ commit pipeline.

### TypeScript (SDK, CLI, MCP)

1. Write generated files to temp dir
2. Run `tsc --noEmit --strict` against the temp dir
3. If `tsc` exits non-zero ŌåÆ mark that generator as errored, log tsc output, exclude from commit
4. Run `prettier --write "**/*.ts"` on passing files
5. Commit formatted files

### Python (SDK)

1. Write generated files to temp dir
2. Run `ruff check --fix .` (auto-fix safe issues)
3. Run `ruff format .`
4. If ruff exits non-zero on check (unfixable errors) ŌåÆ log warning but do not fail the run (generated Python is rarely unsalvageable)
5. Commit formatted files

### Rules

- Temp dir created per run under `/tmp/emitkit-{runId}/`
- Temp dir cleaned up after job completes (success or failure)
- `tsc` and `prettier` must be available in the worker environment (installed as dev deps in `apps/worker`)
- `ruff` installed as a Python tool available on PATH in worker environment
- Syntax check failures are per-generator ŌĆö one failing generator does not block others

## 9. SDK Version Control

### Version Strategy: `emitkit-managed` (default)

Emitkit calculates the next semver based on a diff of the current spec against `generation_runs.specSnapshot` from the last successful run.

| Change detected                                | Bump    |
| ---------------------------------------------- | ------- |
| Operation removed                              | `minor` |
| Required parameter added to existing operation | `minor` |
| Required field added to response schema        | `minor` |
| New operation added                            | `patch` |
| Description / summary change only              | `patch` |
| `info.version` major segment changed in spec   | `major` |
| First run (no prior snapshot)                  | `0.1.0` |

- Calculated version written into `package.json`, `pyproject.toml`, MCP server `package.json`, and docs footer
- Stored in `generation_runs.sdkVersion` and a new `sdk_versions` row
- Auto-generated changelog: list of added/removed/changed operation IDs in markdown

### Version Strategy: `spec-version`

- Read `info.version` directly from OpenAPI spec
- Use it as-is for all outputs
- No diff calculation needed
- Stored the same way

### Git Tagging (post-merge)

- On `pull_request` closed + merged webhook event:
  - Create Git tag `sdk/v{version}` on the merge commit
  - Create Git tag `mcp/v{version}` if MCP was generated
- Tags created via GitHub API (`POST /repos/{owner}/{repo}/git/refs`)

## 10. Custom Method Injection

The core problem: generated files are overwritten on every run. Custom methods must survive regeneration.

### Protected Files Pattern

- Any file under `{outputDir}/sdk/typescript/custom/` or `{outputDir}/sdk/python/custom/` is **never overwritten**
- `OutputFile.skipIfExists = true` for all custom stub files
- On first run: generate stub files in `custom/` (empty extension class/module)
- On subsequent runs: skip writing `custom/` files entirely (check file existence via GitHub API before committing)

### TypeScript Example

```ts
// sdk/typescript/resources/payments.ts (GENERATED ŌĆö overwritten each run)
import { PaymentsCustom } from '../custom/payments'
export class Payments extends PaymentsCustom {
  async create(body: CreatePaymentRequest): Promise<Payment> { ... }
}

// sdk/typescript/custom/payments.ts (STUB ŌĆö written once, never overwritten)
import { BaseClient } from '../client'
export class PaymentsCustom extends BaseClient {
  // Add custom methods here ŌĆö they survive regeneration
}
```

### Python Example

```python
# sdk/python/resources/payments.py (GENERATED)
from ..custom.payments import PaymentsCustom
class Payments(PaymentsCustom):
    def create(self, body: CreatePaymentRequest) -> Payment: ...

# sdk/python/custom/payments.py (STUB ŌĆö never overwritten)
from ..client import BaseClient
class PaymentsCustom(BaseClient):
    # Add custom methods here
    pass
```

### Checking File Existence Before Commit

- Before building the GitHub tree for commit, check which `custom/` files already exist in the repo
- Use `GET /repos/{owner}/{repo}/contents/{path}` for each custom file
- If file exists ŌåÆ exclude from tree (preserve existing content)
- If file does not exist ŌåÆ include stub in tree (first run)

## 11. SDK Publishing

Emitkit delegates publishing to the repo's own CI. It generates the workflow files as part of each run.

### TypeScript ŌåÆ npm

Generated file: `.github/workflows/publish-npm.yml`

```yaml
name: Publish to npm
on:
  pull_request:
    types: [closed]
    branches: [main]
jobs:
  publish:
    if: github.event.pull_request.merged == true && startsWith(github.head_ref, 'emitkit/')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          registry-url: "https://registry.npmjs.org"
      - run: cd .emitkit/sdk/typescript && npm ci && npm publish
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

Admin sets `NPM_TOKEN` as a repo secret once. Emitkit does not touch it.

### Python ŌåÆ PyPI

Generated file: `.github/workflows/publish-pypi.yml`

```yaml
name: Publish to PyPI
on:
  pull_request:
    types: [closed]
    branches: [main]
jobs:
  publish:
```

## 13. GitHub Integration

### OAuth Scopes Required

```
repo          (read/write to private and public repos)
read:org      (list org memberships)
workflow      (write GitHub Actions workflow files)
```

### Webhook Setup

- On project creation, register a `push` + `pull_request` webhook on the repo via GitHub API
- Webhook URL: `{EMITKIT_BASE_URL}/webhooks/github`
- Webhook secret: random 32-byte hex, stored encrypted per project
- Events: `push`, `pull_request`

### Spec Fetching

- `GET /repos/{owner}/{repo}/contents/{path}?ref={sha}`
- Decode base64 content
- Validate as valid YAML or JSON before passing to parser
- Reject if spec > 2MB

### New Repo Creation

- `POST /user/repos` or `POST /orgs/{org}/repos`
- Create initial commit with starter `openapi.yaml`:
  ```yaml
  openapi: 3.1.0
  info:
    title: My API
    version: 0.1.0
  paths: {}
  ```
- Register webhook on the new repo immediately after creation

### Committing Output

1. Determine output repo (same repo or `outputRepoFullName` if `outputMode = separate`)
2. Get current tree SHA of default branch
3. Check which `custom/` files already exist (to preserve them)
4. Create blobs for all new/updated files
5. Create new tree with blobs (exclude existing custom files)
6. Create commit: `chore: emitkit run #${runId} ŌĆö ${sdkVersion}`
7. Create branch `emitkit/run-{runId}`
8. Create PR: base = default branch, head = new branch
9. PR body includes: outputs generated, languages, SDK version, spec commit SHA, changelog

### Rate Limiting

- All Octokit calls wrapped in retry wrapper (3 attempts, respect `Retry-After` on 429)
- Log each GitHub API call (method + path) in run logs

## 15. Backend Rules & Constraints

### 15.1 oRPC Procedures

- Every procedure validates input with Zod
- Every procedure checks session; throw `ORPCError('UNAUTHORIZED')` if not authenticated
- Every procedure checks org membership before operating on org resources; throw `ORPCError('FORBIDDEN')` if not a member
- Procedures do not directly import Octokit; they call service functions from `packages/github`
- Procedures do not directly run generators or enqueue jobs; they call service functions from `apps/server/src/services/`

### 15.2 Service Layer

- Business logic lives in `apps/server/src/services/`
- Services are plain async functions; no classes
- DB access only via Drizzle ŌĆö never raw SQL strings
- Each service function handles one responsibility
- Services enqueue BullMQ jobs via `packages/queue`

### 15.3 BullMQ Job Definitions

```ts
// packages/queue/index.ts

export const QUEUES = {
  GENERATION: "generation",
} as const;

export interface GenerationJobData {
  runId: string;
}

export interface GenerationJobResult {
  prUrl?: string;
  sdkVersion?: string;
}
```

- Queue defined in `packages/queue`; both server and worker import from here
- Worker in `apps/worker/src/index.ts` ŌĆö separate Bun process started independently
- Job options: `attempts: 2`, `backoff: { type: 'exponential', delay: 5000 }`, `timeout: 300000` (5 min)
- Failed jobs (exhausted retries): mark run as `failed` in DB, publish `[DONE]` to Redis SSE channel

### 15.4 Security

- GitHub OAuth tokens encrypted at rest (AES-256-GCM, key from `ENCRYPTION_KEY` env var)
- Gemini API keys encrypted at rest (same key)
- Webhook secrets encrypted at rest (same key)
- Webhook payloads verified with HMAC-SHA256 before processing
- All user inputs sanitized via Zod before DB writes
- No SQL interpolation ŌĆö Drizzle parameterized queries only
- Secrets never logged

### 15.5 Logging

- Structured JSON logs via `pino`
- Log levels: `error`, `warn`, `info`, `debug`
- Each request logged with method, path, status, duration
- Never log: access tokens, API keys, webhook secrets, session tokens

### 15.6 Error Responses

| Scenario                            | oRPC Error Code         | HTTP Status |
| ----------------------------------- | ----------------------- | ----------- |
| Not authenticated                   | `UNAUTHORIZED`          | 401         |
| Not org member                      | `FORBIDDEN`             | 403         |
| Resource not found                  | `NOT_FOUND`             | 404         |
| Validation failure                  | `BAD_REQUEST`           | 400         |
| GitHub API failure                  | `INTERNAL_SERVER_ERROR` | 500         |
| Spec parse failure                  | `UNPROCESSABLE_CONTENT` | 422         |
| Spec SHA conflict (concurrent edit) | `CONFLICT`              | 409         |

## 18. Error Handling Contract

### Generator Errors

- Each generator wraps its logic in try/catch
- On error: return `{ files: [], error: string }`
- Runner collects results per generator; partial success is allowed
  - e.g. Python SDK fails tsc check, TypeScript SDK succeeds ŌåÆ commit TS output, log Python failure as warning
- If ALL generators fail and no files are produced: mark run `failed`, no PR created

### Syntax Check Errors

- TypeScript (`tsc`) failure on a generator's output: exclude that generator's files from commit, log tsc output
- Python (`ruff`) unfixable errors: log as warning, still commit (ruff format output is best-effort)
- Syntax check failures never block other generators

### GitHub API Errors

- 401/403: mark run `failed`, log "GitHub auth error ŌĆö re-authenticate"
- 404 on spec fetch: mark run `failed`, log "Spec file not found at {path}"
- 409 on branch create: append `-{timestamp}` suffix and retry once
- All other errors: mark run `failed`, log full error message

### Spec Parsing Errors

- Invalid YAML/JSON: mark run `failed`, log parse error with line number if available
- OpenAPI validation errors: mark run `failed`, log all validation errors
- Spec > 2MB: mark run `failed`, log "Spec file exceeds 2MB limit"

### BullMQ Job Failures

- Job throws unexpectedly: BullMQ retries up to 2 times (exponential backoff)
- After exhausted retries: mark run `failed`, publish `[DONE]` to Redis SSE channel
- Stale `running` runs on worker startup: marked `failed` immediately

## 19. Out of Scope

The following are explicitly deferred and should not be built:

- Go, Ruby, Java SDK output ŌĆö TypeScript and Python only
- Spec diffing at the field level (only operation-level diff for version bumping)
- Billing / plan limits
- Public REST API for external triggers (beyond GitHub webhook)
- Terraform provider generation
- Self-hosted deployment tooling (Docker, Helm charts)
- Rate limit dashboards or usage analytics
- Email / Slack notifications on run completion
- SDK playground / interactive docs
